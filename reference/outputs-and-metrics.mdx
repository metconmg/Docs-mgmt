Every MCMG run produces a structured set of output files that capture different aspects of the simulation at different levels of granularity. Understanding what each output type contains, how it is formatted, and how to read it is essential for analysing results, diagnosing problems, and feeding data into downstream tools. This reference documents every output type produced by the engine, the fields each record contains, and how to work with the files after a run completes.

## Output Types

MCMG produces four distinct categories of output during a simulation run. Each is written to its own subdirectory beneath the configured `output.directory`.

| Output Type | Subdirectory | Description |
|---|---|---|
| Event Log | `events/` | Append-only record of every discrete event processed during the run |
| State Snapshot | `snapshots/` | Periodic full-state captures of all live entities |
| Summary Report | `reports/` | End-of-run aggregate metrics and run metadata |
| Error Log | `errors/` | Structured record of warnings and errors emitted during the run |

---

## Event Log

The event log is the most granular output MCMG produces. It records every event that enters and exits the event queue, giving you a complete causal history of the simulation. By default it is written in JSONL format (one JSON object per line), making it straightforward to stream, grep, or import into data tools.

### File Naming

Event log files are named using the pattern:

```
events_<run_id>_part<N>.<format>
```

Where `<run_id>` is the unique identifier for the run, `<N>` is a zero-padded sequence number that increments when the file is rotated, and `<format>` is the configured output format (`jsonl`, `csv`, or `parquet`).

### Event Log Fields

<ResponseField name="event_id" type="string">
  Unique identifier for this event, generated by the engine at the time the event is enqueued. Format is a hex-encoded 128-bit value. Stable across replays of the same seed and configuration.
</ResponseField>

<ResponseField name="timestamp" type="float">
  The simulation time (in STUs) at which this event was scheduled to fire. For instantaneous events this is identical to `processed_at`; for delayed events the two values differ.
</ResponseField>

<ResponseField name="processed_at" type="float">
  The simulation time at which the engine actually processed this event. May differ from `timestamp` when events are batched or when lookahead buffering is enabled.
</ResponseField>

<ResponseField name="event_type" type="string">
  A namespaced string identifying the kind of event. Built-in types use the `engine.` prefix; user-defined event types use the `scenario.` prefix by convention. Examples: `engine.entity_created`, `engine.entity_destroyed`, `scenario.interaction_started`.
</ResponseField>

<ResponseField name="priority" type="integer">
  Scheduling priority of the event. Higher values are processed before lower values when events share the same timestamp. Range: `1` to `global.max_event_priority`.
</ResponseField>

<ResponseField name="source_entity_id" type="string">
  The ID of the entity that generated this event, or `null` for system-generated events.
</ResponseField>

<ResponseField name="target_entity_id" type="string">
  The ID of the entity this event is directed at, or `null` for broadcast or environmental events.
</ResponseField>

<ResponseField name="payload" type="object">
  A free-form JSON object carrying event-specific data. The keys and value types depend on the event type. Consult the event type registry for schema details for each event type your scenario uses.
</ResponseField>

<ResponseField name="run_id" type="string">
  The unique run identifier, repeated in every record for convenience when multiple runs are merged into a single dataset.
</ResponseField>

### Example Event Log Record

```json
{
  "event_id": "3f7a2c1e9b004d8eab12c3f445d6e789",
  "timestamp": 12.4,
  "processed_at": 12.4,
  "event_type": "scenario.interaction_started",
  "priority": 50,
  "source_entity_id": "ent_00042",
  "target_entity_id": "ent_00107",
  "payload": {
    "interaction_type": "proximity",
    "distance": 0.82,
    "duration_estimate": 3.0
  },
  "run_id": "run_20240815_001"
}
```

## State Snapshot

A state snapshot is a complete serialised image of the simulation world at a given point in time. Snapshots allow you to reconstruct the state of every entity without replaying the entire event log. They are produced at the interval configured by `output.snapshot_interval`.

### File Naming

Snapshot files are named:

```
snapshot_<run_id>_t<timestamp_padded>.<format>
```

Where `<timestamp_padded>` is the snapshot simulation time formatted as a zero-padded decimal with two fractional digits (for example, `t00240.00` for a snapshot taken at STU 240.0).

### When Snapshots Are Produced

The engine writes a snapshot:
- At simulation time `0.0` (the initial state, regardless of `snapshot_interval`)
- Every `output.snapshot_interval` STUs thereafter
- At the final simulation time when the run ends normally
- Immediately before any controlled shutdown triggered by a fatal error

### Snapshot Record Fields

Each snapshot file contains a top-level metadata object followed by an array of entity records.

**Snapshot metadata fields:**

<ResponseField name="snapshot_id" type="string">
  Unique identifier for this snapshot.
</ResponseField>

<ResponseField name="run_id" type="string">
  Identifier of the run that produced this snapshot.
</ResponseField>

<ResponseField name="simulation_time" type="float">
  The simulation clock value at the moment the snapshot was captured.
</ResponseField>

<ResponseField name="tick" type="integer">
  The engine tick counter at the time of capture.
</ResponseField>

<ResponseField name="entity_count" type="integer">
  Number of live entities captured in this snapshot.
</ResponseField>

<ResponseField name="snapshot_type" type="string">
  One of `"scheduled"`, `"initial"`, `"final"`, or `"emergency"`. Indicates why this snapshot was taken.
</ResponseField>

**Entity record fields (when `output.include_entity_detail` is `true`):**

<ResponseField name="entity_id" type="string">
  Unique identifier for this entity.
</ResponseField>

<ResponseField name="archetype" type="string">
  The entity archetype (type definition) this entity was created from.
</ResponseField>

<ResponseField name="position" type="array">
  Spatial coordinates as a two- or three-element float array, matching `global.spatial_dimensions`.
</ResponseField>

<ResponseField name="velocity" type="array">
  Current velocity vector, same dimensionality as `position`.
</ResponseField>

<ResponseField name="attributes" type="object">
  Key-value map of all current attribute values for this entity. Keys are the attribute names defined in the archetype plus any dynamically added attributes.
</ResponseField>

<ResponseField name="lifecycle_state" type="string">
  Current lifecycle state of the entity. Common values: `"active"`, `"dormant"`, `"spawning"`, `"destroying"`.
</ResponseField>

## Summary Report

The summary report is produced once at the end of each run. It provides a concise overview of what happened, aggregate metrics, and the provenance information needed to reproduce or cite the run. The report format is controlled by `output.summary_report.format`.

### Always-Included Metrics

The following metrics appear in every summary report regardless of scenario or configuration:

| Metric | Type | Description |
|---|---|---|
| `run_id` | string | Unique run identifier |
| `scenario_name` | string | Value of `scenario.name` |
| `scenario_version` | string | Value of `scenario.version` |
| `seed` | integer | Effective seed used (after any run-level override) |
| `fidelity` | string | Effective fidelity level |
| `wall_clock_start` | ISO-8601 datetime | Real-world timestamp when the run began |
| `wall_clock_end` | ISO-8601 datetime | Real-world timestamp when the run ended |
| `wall_clock_duration_s` | float | Real-world elapsed time in seconds |
| `simulation_duration_stu` | float | Total simulated time covered |
| `ticks_completed` | integer | Number of engine ticks executed |
| `total_events_processed` | integer | Cumulative events processed from the queue |
| `entity_peak_count` | integer | Highest simultaneous entity count reached during the run |
| `entity_total_created` | integer | Total entities created over the full run |
| `entity_total_destroyed` | integer | Total entities destroyed over the full run |
| `error_count` | integer | Number of error-level log records emitted |
| `warning_count` | integer | Number of warn-level log records emitted |
| `snapshots_written` | integer | Number of state snapshots written |
| `output_size_bytes` | integer | Combined size of all output files in bytes |

## Key Performance Metrics

Beyond the always-included report fields, MCMG tracks several runtime performance metrics that help you understand whether the simulation is running efficiently and whether results are reliable.

### Throughput

**Throughput** measures how many simulation events the engine processes per second of real-world wall-clock time.

```
throughput (events/sec) = total_events_processed / wall_clock_duration_s
```

Throughput varies throughout a run depending on entity population and event density. The summary report provides the overall average; enabling profiling (`run.profile: true`) gives you per-interval throughput data.

### Entity Peak Count

The **entity peak count** is the highest number of simultaneously live entities at any tick. It is the primary signal for memory and CPU demand: if `entity_peak_count` approaches `engine.max_entities`, consider increasing the limit or adjusting spawn rates.

### Memory High-Water Mark

The **memory high-water mark** is the maximum resident set size (RSS) observed during the run, in megabytes. It is included in the summary report when profiling is enabled and in the error log if the engine detects memory pressure. Use this value to size memory allocations when planning batch runs on shared infrastructure.

## Error Log

The error log captures every warning and error the engine emits during a run. It is separate from the primary simulation log (which records operational messages) and is intended as a focused audit trail for problems.

Error log records use the same JSONL format as event logs. Each record includes:

<ResponseField name="severity" type="string">
  One of `"warn"`, `"error"`, or `"fatal"`. Fatal records indicate conditions that caused the run to halt.
</ResponseField>

<ResponseField name="simulation_time" type="float">
  The simulation clock value when the condition was detected.
</ResponseField>

<ResponseField name="code" type="string">
  A stable error code string. Codes do not change across engine versions, making them safe to match programmatically.
</ResponseField>

<ResponseField name="message" type="string">
  A human-readable description of the condition.
</ResponseField>

<ResponseField name="context" type="object">
  Structured key-value pairs providing additional context, such as the entity ID, parameter name, or event ID associated with the condition.
</ResponseField>

## Output File Naming Conventions

All output filenames follow a consistent pattern to support automated processing:

```
<type>_<run_id>[_<qualifier>].<format>
```

- **`<type>`**: One of `events`, `snapshot`, `summary`, `errors`.
- **`<run_id>`**: The unique run identifier (e.g., `run_20240815_001`).
- **`<qualifier>`**: Optional segment providing additional context, such as a timestamp for snapshots or a part number for rotated event logs.
- **`<format>`**: File extension matching the output format (`jsonl`, `csv`, `parquet`, `json`, `yaml`, or `txt`).

Run identifiers are generated from the real-world date, an incrementing counter, and the scenario name hash, ensuring lexicographic sort order matches chronological order when runs share the same date prefix.

## Exporting Outputs to External Tools

MCMG outputs are designed to be portable. The following patterns cover the most common export workflows.

### Loading JSONL Event Logs

Most data processing environments can read JSONL directly:

```python
import json

records = []
with open("output/events/events_run_20240815_001_part0.jsonl") as fh:
    for line in fh:
        records.append(json.loads(line))
```

### Converting to a Columnar Format

If you configured `parquet` as an output format, the files are directly readable by columnar query engines. For CSV files, standard import functionality in any data tool applies.

### Merging Multiple Runs

Because every record carries a `run_id` field, you can concatenate event logs or snapshot files from multiple runs into a single dataset and use `run_id` as a grouping key for comparative analysis. Ensure all runs used compatible scenario versions before merging.

### Replaying a Run from Snapshots

To resume or replay a run from a specific snapshot, set `scenario.initial_state_file` to the path of the desired snapshot file and set `scenario.seed` to the same seed used in the original run. The engine will initialise entity state from the snapshot and continue from the snapshot's `simulation_time`.
